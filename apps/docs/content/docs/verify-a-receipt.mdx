---
title: Verify a receipt
description: How a third party can confirm a Nexus signed-inference receipt is real.
---

A signed-inference receipt is only useful if downstream actors can
*verify* it without trusting the agent that received it. Here's how.

## What you can verify locally

Given a receipt + the prompt + the response that were claimed:

1. **The prompt is what the agent says it is.** Recompute
   `sha256(role:content concatenated by newline)` over the messages and
   compare to `receipt.prompt_hash`.
2. **The response is what the agent says it is.** Recompute
   `sha256(response_text)` and compare to `receipt.response_hash`.
3. **The receipt format hasn't been tampered with.** Today the receipt
   is a structured object, not yet Ed25519-signed by Nexus — that's
   coming. For now, the verification chain is:
   - Prompt + response hashes match the claimed values
   - Payment `tx_signature` exists on Solana and pays the claimed amount
     to the claimed recipient

## What you can verify on-chain

The `receipt.payment.tx_signature` is the strongest signal — it's a
Solana transaction that anyone can fetch:

```ts
import { Connection } from "@solana/web3.js";

const conn = new Connection("https://api.devnet.solana.com");
const tx = await conn.getTransaction(receipt.payment.tx_signature, {
  maxSupportedTransactionVersion: 0,
});

// Verify:
// - Status is Success
// - One USDC transfer instruction is present
// - The transfer source ATA is owned by receipt.agent_pubkey
// - The transfer dest ATA is owned by the expected Nexus deposit address
// - The amount matches receipt.payment.amount_usdc (× 1_000_000 atomic)
```

If all four checks pass, you know:

- The agent claimed by the receipt actually paid the claimed amount of
  USDC on-chain
- The Solana settlement timestamp is roughly when the receipt says it is
  (`receipt.timestamp` is server-set; the on-chain `blockTime` is
  authoritative)

## What's NOT yet verifiable

Today there's no Ed25519 signature by Nexus over the receipt. That
means if you don't trust the entity that handed you the receipt, you
can't *prove* Nexus issued it (only that the payment happened).

The fix is to sign the receipt JSON with a Nexus operator key and
publish the public key alongside the SDK. **That ships in the
post-v0.1 hardening pass** — tracked as a roadmap item.

For agent-to-agent flows where both ends trust Nexus, the current
hash-based receipt + on-chain payment is sufficient: the receiving
agent independently fetches the tx, recomputes hashes, and acts.

## Reference verifier

A runnable verifier ships in the SDK starting in v0.2:

```ts
// (preview — not yet released)
import { verifyReceipt } from "@vdm-nexus/x402/verify";

const result = await verifyReceipt({
  receipt,
  prompt: originalMessages,
  response: replyText,
  rpc: "https://api.devnet.solana.com",
});

// result: {
//   prompt_hash_ok: boolean,
//   response_hash_ok: boolean,
//   payment_on_chain: boolean,
//   payer_matches: boolean,
//   amount_matches: boolean,
// }
```

Until v0.2 ships, the manual checks above are the path.
